# Feature Spec Template

## Summary
One-paragraph description of the feature and why it matters.

## Goals
- Primary outcome(s) this feature must achieve.
- User/business value tied to those outcomes.

## Non-Goals
- Explicitly list what is out of scope.

## Background / Context
- Relevant history, dependencies, constraints, or references.

## User Stories
- As a <role>, I want <capability> so that <benefit>.

## Requirements
### Functional
- List required behaviors and flows.

### Non-Functional
- Performance, availability, security, compliance, latency, cost, etc.

## UX / Flow
- Key screens, states, and transitions.
- Wireframes/mock links if available.

## API / Interfaces
- New/changed endpoints, events, contracts, or schemas.
- Input/output examples if useful.

## Data Model
- New/changed entities, fields, indexes, migrations.

## On-Chain (if applicable)
- Contracts, methods, events, gas considerations.

## Architecture & Components
- Affected services, modules, or boundaries.
- High-level sequence or diagram references.

## Observability
- Logs, metrics, traces, alerts.
- SLOs or key dashboards.

## Security & Privacy
- Threat model notes, authz/authn, data handling.

## Rollout Plan
- Feature flags, migration steps, backwards compatibility.

## Testing Plan
- Unit, integration, e2e, contract tests.

## Risks & Mitigations
- Known risks and how to reduce them.

## Open Questions
- Decisions needed before implementation.

## Acceptance Criteria
- Clear, testable conditions for completion.
